# TMS Web Service アーキテクチャ

## プラットフォーム全体像

Agentic QA プラットフォームは4つの独立したサブシステムから構成される。TMS Web Service（本リポジトリ）は**ハブ**として機能し、他3つは疎結合な**衛星サービス**として TMS の公開 API を叩く。

```
┌─────────────────────────────────────────────────────────────────┐
│                    Agentic QA プラットフォーム                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Test Case Discovery]     衛星 — コード解析による仕様抽出        │
│          │                                                      │
│          ▼                                                      │
│  ┌─────────────────────┐                                        │
│  │  TMS Web Service    │◀── ★ 本設計の対象（ハブ）              │
│  │  構造化テストケースの │                                        │
│  │  管理・仕様マスター   │                                        │
│  └─────────────────────┘                                        │
│          ▲         ▲                                            │
│          │         │                                            │
│  [Agentic Test Gen]   [Self-Healing]                            │
│  衛星 — テストコード生成  衛星 — CI連携自動修復                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**設計の要点:** TMS 本体から LLM・コード解析の複雑さを完全排除し、枯れた技術で作る CRUD Web サービスに徹する。構造化テストケースのデータモデルが全衛星サービスの統一契約となる。

---

## 本設計のスコープ

**対象:** TMS Web Service 単体（ハブ）— テストケースの CRUD・取り込み API・簡易履歴・最小認証・素朴 UI。

**対象外:** Discovery / Test Generation / Self-Healing の中身。本設計では衛星サービス向けの「取り込み API の口」のみ定義する（→ [sync-protocol.md](sync-protocol.md)）。

---

## 主要な意思決定

| # | 論点 | 決定 | 理由 |
|---|------|------|------|
| 1 | 成果物のゴール | 汎用 SaaS を見据えつつ、最初の対象は Java/JUnit 単体テストに限定 | 多言語抽象化に溺れず縦に1本通す |
| 2 | 最初のサブシステム | TMS Web Service（ハブ） | 全衛星の契約となるデータモデルを先に固める |
| 3 | テストケースの保持形式 | 構造化スキーマ（Gherkin/自然言語は派生ビュー） | 差分比較・自己修復は構造化前提 |
| 4 | TMS MVP 範囲 | CRUD＋取り込み API＋簡易履歴＋最小認証＋素朴 UI（HITL 承認ワークフローは初版対象外） | シンプルさ優先 |
| 5 | 実行基盤 | Cloudflare Workers | 利用者指定 |
| 6 | 技術スタック | Workers + Hono + D1 + Drizzle ORM + Zod + Hono JSX(SSR) | サーバーレスで CRUD＋API＋UI を1アプリで完結 |
| 7 | オンプレ移植性 | 第一級要件。CF 固有機能はアダプタ裏に隔離 | クラウド/オンプレ差分を最小化 |
| 8 | 実装言語 | TypeScript 据え置き | CRUD は I/O バウンド。言語変更の体感改善はほぼゼロ |

## レイヤ構成

```
┌──────────────────────────────────────────────────────────────┐
│                    TMS Web Service                            │
├──────────────────────────────────────────────────────────────┤
│  UI 層（Hono JSX）          … HTML 描画のみ                    │
│  API 層（Hono ルート）       … JSON 入出力・Zod 検証            │
├──────────────────────────────────────────────────────────────┤
│  ドメイン層                  … ビジネスロジック（HTTP も DB も知らない）│
├──────────────────────────────────────────────────────────────┤
│  Storage / Auth / RateLimiter インターフェース                  │
├──────────────────────────────────────────────────────────────┤
│  アダプタ実装                                                  │
│  ┌──────────┐  ┌───────────────┐  ┌──────────────────────┐  │
│  │ D1       │  │ libSQL        │  │ better-sqlite3       │  │
│  │(Workers) │  │(オンプレ)      │  │(オンプレ)             │  │
│  └──────────┘  └───────────────┘  └──────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

| レイヤ | 責務 | 依存先 |
|--------|------|--------|
| **UI 層** | HTML 描画のみ。ドメイン層を呼ぶ | ドメイン層 |
| **API 層** | JSON 入出力・Zod 検証・HTTP 関心事のみ | ドメイン層 |
| **ドメイン層** | テストケース/プロジェクト/履歴のビジネスロジック | Storage インターフェース |
| **Storage インターフェース** | `getTestCases()`, `createTestCase()` 等の抽象 | （実装非依存） |
| **アダプタ実装** | D1 / libSQL / better-sqlite3 ごとの具体実装（Drizzle で共通化） | Drizzle + 各ドライバ |

**原則:** UI 層・API 層はドメイン層しか知らない。ドメイン層は Storage インターフェースしか知らない。CF 固有のバインディングはアダプタの内側に閉じる。

## ポータビリティ境界

アプリ本体（UI/API/ドメイン）が参照するインターフェースは3つ＋1つ：

| インターフェース | 役割 | CF 実装 | オンプレ実装 |
|------------------|------|---------|-------------|
| **Storage** | データアクセス抽象 | D1 アダプタ（Drizzle） | libSQL / better-sqlite3 アダプタ（Drizzle） |
| **Auth** | 認証ロジック抽象 | WebCrypto PBKDF2 | argon2 等の別 IdP 差し替え可 |
| **RateLimiter** | 流量制御抽象 | Workers Rate Limiting binding | プロセス内 / Redis アダプタ |
| *(ランタイム設定)* | Smart Placement | `placement = { mode = "smart" }` | 設定不在（無害） |

**差分は「起動エントリ＋注入するアダプタ＋環境設定」のみ。** ビジネスロジック・UI・API は 100% 共通。

### クラウド版エントリ

- Cloudflare Workers
- D1 アダプタ
- （オプション）KV / Cache API アダプタ（読みキャッシュ）
- Workers Rate Limiting binding

### オンプレ版エントリ

- Node.js / Bun サーバー
- libSQL or better-sqlite3 アダプタ
- プロセス内 / Redis レートリミッタ
- `PRAGMA auto_vacuum=INCREMENTAL` を接続初期化フックで発行

## UI 方針

Hono JSX の SSR ＋ HTMX で部分更新。最小 JS。

主要画面は3つ：

1. **プロジェクト一覧/作成**
2. **テストケース一覧**（フィルタ・検索・HTMX 部分更新）
3. **テストケース詳細/編集**（構造化フォーム・Gherkin ビュー・変更履歴・drift/stale バッジ・構造化 Diff）

画面の詳細定義・遷移図は → [screens.md](screens.md)
ユーザー操作フローは → [usecase.md](usecase.md)

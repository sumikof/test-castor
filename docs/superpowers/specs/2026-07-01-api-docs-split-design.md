# API ドキュメント分割設計

## 目的

モノリシックな `docs/api-reference.md`（1085行）を、APIグループ単位の個別ドキュメントに分割する。`api-reference.md` は索引 + 共通仕様として残す。

目標は「コードではなく仕様を明確にする」こと。実装の詳細ではなく、各APIの入出力・業務ルール・エラー条件を正確に記述する。

## 決定事項

| 項目 | 決定 |
|---|---|
| 分割粒度 | APIグループ単位（6ファイル） |
| ディレクトリ構造 | `docs/apis/` 直下にフラット配置 |
| JSONサンプル | テーブルで仕様定義 + 代表的なJSON例を併用 |
| 共通仕様の配置 | `api-reference.md` に残留 |

## ファイル構成

```
docs/
├── api-reference.md          ← 索引 + 共通仕様（既存を改修）
└── apis/
    ├── setup.md              ← 初期セットアップ（1エンドポイント）
    ├── auth.md               ← 認証（5エンドポイント）
    ├── users.md              ← ユーザー管理（5エンドポイント）
    ├── projects.md           ← プロジェクト（3エンドポイント）
    ├── testcases.md          ← テストケース（12エンドポイント）
    └── tokens.md             ← APIトークン管理（3エンドポイント）
```

## `api-reference.md` 改修後の構成

1. **このドキュメントについて** — 導入文（既存維持）
2. **共通仕様** — ベースパス、IDOR防止、日時表現、ページング、バージョニング規約（既存維持）
3. **統一エラースキーマ** — エラーコード一覧（既存維持）
4. **OCC / PATCHセマンティクス / 冪等性 / origin正規化** — 既存維持
5. **エンドポイント一覧** — グループごとのテーブル + 各詳細ドキュメントへのリンク（個別詳細は削除しリンクで置換）
6. **関連ドキュメント** — 既存リンク集を維持

## 各APIドキュメントのテンプレート

各ファイルは以下のセクションで統一する：

### 1. 概要

このAPIグループが何を担うか、1〜2文で説明。

### 2. エンドポイント一覧テーブル

| メソッド | パス | 用途 | 認証 |
|---|---|---|---|

### 3. 各エンドポイントの詳細

エンドポイントごとに以下のサブセクションを記載：

- **説明** — 何をするAPIか、1〜2文
- **認証・認可** — 必要なロール、認証方式
- **リクエスト仕様** — フィールド定義テーブル（名前 / 型 / 必須 / 説明）
- **レスポンス仕様** — ステータスコード別にフィールド定義テーブル
- **リクエスト例・レスポンス例** — 代表的なJSONを1つずつ
- **エラー** — 発生しうるエラーコードとその条件
- **副作用・業務ルール** — ownership遷移、セッション無効化など、そのエンドポイント固有の振る舞い

### 4. 共通仕様への参照

OCC・ページング等が適用される場合は `api-reference.md` への参照リンクを記載。

## 移行ルール

- 各エンドポイントの詳細仕様（リクエスト/レスポンス/エラー/副作用）は個別ドキュメントに移動
- 共通仕様セクション（ベースパス〜origin正規化）は `api-reference.md` に残留
- エンドポイント一覧テーブルは `api-reference.md` に残し、各詳細ドキュメントへのリンクを付与
- 情報の欠落がないことを移行後に確認する

## パターンの一貫性

`screens.md`（索引 + 画面遷移図）→ `screens/` ディレクトリ（個別画面ドキュメント）のパターンと同様の構造を採用。プロジェクト内のドキュメント構造の一貫性を維持する。

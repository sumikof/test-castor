# TMS ユースケース — テスト管理の実践ガイド

## このドキュメントについて

開発中のシステムに TMS（Test Case Management System）を導入し、テストを管理するまでの一連のフローを記述する。「最初に何をするか」「次に何をするか」を時系列で追い、各場面でユーザーが行う操作を具体的に示す。

現設計スペックに未定義の機能は `[※未実装]` で明示する。多くは設計スペックへの反映が完了しており（→ ギャップ一覧）、残る `[※未実装]` は主に MVP 後の拡張機能（ダッシュボード、レポート、テストレベル分類等）に限定される。

### 対象読者

| ロール | 主な関心 | 権限 |
|---|---|---|
| 管理者 (admin) | システム運用、アクセス管理 | 全操作 + ユーザー管理 + トークン管理 |
| 編集者 (editor) | テストケースの作成・編集・承認 | テストケース CRUD + ステータス変更 |
| 閲覧者 (viewer) | テスト状況の確認 | 読み取りのみ |

## フェーズ 1：導入と初期設定

### UC-01 システムのデプロイとアクセス

管理者が TMS をデプロイし、アクセス可能にする。

1. クラウド版：Cloudflare Workers にデプロイ（`wrangler deploy`）
2. オンプレ版：Node.js / Bun サーバーとして起動
3. ブラウザで TMS の URL にアクセスできることを確認

### UC-02 初期管理者アカウントの作成

デプロイ直後、最初の管理者アカウントを作成する。

1. 初回アクセス時にセットアップ画面が表示される（Organization が 0 件の場合）
2. 組織名を入力（例：`株式会社Example`）
3. 管理者のメールアドレス・パスワード・表示名を入力
4. 管理者アカウントと組織が作成され、ログイン画面に遷移

> API: `POST /api/v1/setup` — Organization 未存在時のみ実行可能。

### UC-03 ログイン

1. メールアドレスとパスワードを入力
2. 認証が成功するとセッションが発行される
3. ダッシュボード `[※未実装]` またはプロジェクト一覧画面に遷移

### UC-04 プロジェクトの作成

テスト管理の単位となるプロジェクトを作成する。通常、1つのプロジェクトは1つのリポジトリまたは1つのサービスに対応する。

**実行者:** 管理者

1. プロジェクト一覧画面で「新規プロジェクト」を選択
2. プロジェクト名を入力（例：`payment-service`）
3. リポジトリ URL を入力（任意。開発者がソースを参照するためのリンク）
4. プロジェクトが作成され、プロジェクト一覧に表示される

### UC-05 チームメンバーの追加

プロジェクトに関わるメンバーにアカウントを作成し、適切なロールを割り当てる。

**実行者:** 管理者

1. ユーザー管理画面を開く
2. メンバーのメールアドレス・表示名・ロール（editor / viewer）を入力
3. 初期パスワードを設定（招待メール送信は `[※未実装]`）
4. メンバーがログインしてプロジェクトにアクセス可能になる

> API: `POST /api/v1/users` でユーザー作成。`GET /api/v1/users` で一覧、`PATCH /api/v1/users/:id` でロール変更。

### UC-06 衛星サービス用 API トークンの発行

Discovery 等の衛星サービスが TMS に接続するための API トークンを発行する。

**実行者:** 管理者

1. プロジェクトの設定画面 `[※未実装: 設定画面の導線]` から API トークン管理を開く
2. トークン名（例：`discovery-ci`）を入力して「発行」を実行
3. **画面に平文トークンが 1 回だけ表示される**。コピーして安全な場所に保存する
4. この画面を閉じると平文は二度と確認できない（TMS はハッシュのみ保存）
5. トークンを衛星サービスの設定ファイルや CI の Secret に保存
6. トークン一覧画面では名前・作成日・最終使用日のみ表示される

> トークンはプロジェクト単位のスコープ。他プロジェクトのデータにはアクセスできない。

## フェーズ 2：テストケースの登録

テストケースの登録方法は **自動取り込み** と **手動登録** の 2 つ。

### UC-07 自動取り込み（Discovery 連携）

既存のコードベースからテスト仕様を AI が自動抽出し、TMS に登録する。初期導入時の主要な登録手段。

**前提:** UC-06 で発行したトークンを Discovery に設定済み

1. Discovery 衛星サービスを対象リポジトリに向けて実行する（CLI または CI パイプライン）
2. Discovery がコードを解析し、テスト仕様を構造化データとして抽出する
3. Discovery が TMS API に同期セッションを開始し、テストケースをチャンク送信し、コミットする
4. TMS 上に新しいテストケースが登録される

**登録直後のテストケースの状態:**

| 属性 | 値 | 意味 |
|---|---|---|
| ステータス | `draft` | 未承認。人間のレビュー待ち |
| 所有権 | `machine` | 機械所有。Discovery が最新の観測で内容を自動更新し続ける |
| カテゴリ | Discovery が判定 | `normal` / `abnormal` / `boundary` / `error_handling` |
| fingerprint | Discovery が付与 | 挙動の指紋。後続の drift 検知に使われる |

**ユーザーの次のアクション → フェーズ 3（レビューと承認）へ**

### UC-08 手動登録

コードに対応しないテスト（要件ベースのテスト、探索的テストの記録、手動テストの仕様化等）を人間が直接作成する。

**実行者:** 編集者

1. テストケース一覧画面で「新規作成」を選択
2. 以下を入力する:

| フィールド | 入力内容 | 例 |
|---|---|---|
| タイトル | テストの意図を簡潔に | 「有効期限切れカードで決済を試みるとエラーが返る」 |
| 対象 (target) | テスト対象のクラス・メソッド（任意） | `com.example.PaymentService#charge` |
| カテゴリ | 正常系 / 異常系 / 境界値 / エラーハンドリング | `abnormal` |
| Given | 事前条件 | 「有効期限が過去のカード情報が登録されている」 |
| When | 操作 | 「そのカードで 1,000 円の決済を実行する」 |
| Then | 期待結果 | 「決済が拒否され、エラーコード CARD_EXPIRED が返る」 |
| パラメータ | データ駆動テストの入力値・期待値（任意） | — |
| メタデータ | タグ等（任意） | `{"tags": ["payment", "card-validation"]}` |

3. テストケースが `draft` / `human` 所有で作成される

**手動作成と自動取り込みの違い:**

| | 手動作成 | 自動取り込み |
|---|---|---|
| 所有権 | 最初から `human` | `machine`（初回編集で `human` に不可逆遷移） |
| fingerprint | なし（drift 未評価） | Discovery が付与（drift 検知可） |
| 自動更新 | されない | machine 所有の間は最新の観測で自動ミラー |

### UC-09 パラメータ化テストの登録

同じテストロジックで複数の入力パターンを検証する場合、パラメータ化テストとして登録する。手動登録・自動取り込みのどちらでも利用可能。

**例：年齢バリデーションのテスト**

- **タイトル:** 年齢フィールドのバリデーション
- **Given:** ユーザー登録フォームが表示されている
- **When:** 年齢フィールドに値を入力して送信する
- **Then:** バリデーション結果が返る
- **パラメータ:**

| name | inputs | expected |
|---|---|---|
| 負の値 | `{"age": -1}` | エラー「0以上を入力してください」 |
| 下限境界 | `{"age": 0}` | 成功 |
| 上限境界 | `{"age": 150}` | 成功 |
| 上限超過 | `{"age": 151}` | エラー「150以下を入力してください」 |
| 非数値 | `{"age": "abc"}` | エラー「数値を入力してください」 |

> Given/When/Then は代表シナリオを記述し、入力のバリエーションは parameters 配列に構造化して持つ。

## フェーズ 3：レビューと承認

Discovery が取り込んだテストケースを人間がレビューし、品質を担保する。TMS の中心的なワークフロー。

### UC-10 取り込まれたテストケースのレビュー

**実行者:** 編集者

1. テストケース一覧で以下のフィルタを適用する:
   - ステータス: `draft`
   - 所有権: `machine`（未レビューの取り込み分を特定）
2. テストケースを 1 つ選んで詳細画面を開く
3. 以下の観点で内容を確認する:
   - **タイトル** がテストの意図を正確に表しているか
   - **Given/When/Then** が第三者に伝わる明確さか
   - **カテゴリ** が正しいか（Discovery の誤判定はよくある）
   - **パラメータ** が網羅的か（境界値や代表値が含まれているか）
   - `[※未実装]` 既存テストケースと意味的に **重複** していないか
4. 問題があれば編集する

> **重要:** machine 所有のテストケースを初めて編集すると、所有権が `machine` → `human` に不可逆遷移する。以後 Discovery が内容を自動更新しなくなる。コード側の変化は `drift` として検知される。

5. 内容に問題がなければステータスを `approved` に変更する

### UC-11 編集せずに承認する場合

Discovery の出力がそのまま十分な品質の場合。

1. テストケース詳細画面でステータスを `approved` に変更
2. 所有権が `machine` → `human` に遷移する（承認は人間の明示的な意思決定であるため）
3. 以後 Discovery が内容を自動更新しなくなる

### UC-12 一括承認

大量のテストケースを効率的に処理する。

**実行者:** 編集者

1. テストケース一覧でフィルタ条件を設定（例：`status=draft, category=normal`）
2. 対象テストケースを一括選択（全選択またはチェックボックス）
3. 一括操作メニューから「承認」を選択
4. 確認ダイアログで件数を確認し、実行
5. 選択されたテストケースが全て `approved` / `human` 所有に変更される

> API: `POST /api/v1/projects/:pid/testcases/bulk` — `approve` / `archive` / `restore` の3アクション対応。1リクエスト最大100件。

### UC-13 テストケースの編集

**実行者:** 編集者

1. テストケース詳細画面で「編集」を選択
2. フィールドを修正する（タイトル、Given/When/Then、カテゴリ、パラメータ等）
3. 保存する
4. 楽観的排他制御（OCC）により、他の人が同時に編集していた場合は 409 エラーが返る。その場合は最新の内容を確認してから再度編集する

## フェーズ 4：コード変更への追従

開発が進むとコードが変わり、テスト仕様との乖離が発生する。TMS はこれを自動検知する。

### UC-14 再同期の実行

コードが変更されたら、Discovery を再実行して TMS と同期する。

1. 開発者がコードを変更し、リポジトリに push する
2. Discovery が実行される（CI パイプラインの一部として自動、または手動）
3. Discovery が新しい同期セッションで最新のテスト仕様を TMS に送信する
4. TMS が commit 時に以下を自動判定する:

| 状況 | TMS の動作 |
|---|---|
| 新しいテストケース | 新規 `draft` / `machine` として登録 |
| 変化なし | 何もしない（変化点のみ記録） |
| 挙動が変化した（human 所有） | `drift` フラグを立てる |
| 今回の同期で出現しなかった | `stale` フラグを立てる（approved は除外） |
| machine 所有で挙動が変化した | canonical を最新の観測で自動更新 |

### UC-15 drift（コード乖離）の検知と対応

`human` 所有のテストケースについて、コード側の挙動がテスト仕様と乖離していることを検知する。

**UIでの見え方:**
- テストケース一覧の該当行に `drift` バッジが表示される
- drift フィルタで絞り込みが可能（テストケース一覧 API の `drift` パラメータ）
- テストケース詳細画面に構造化 Diff ビューが表示される

**対応フロー:**

1. テストケース一覧で drift バッジのあるテストケースを開く
2. 構造化 Diff ビューで「現在のテスト仕様（canonical）」と「コードから抽出した最新の観測」の差分を確認する

   ```
   Given:
   - カート内に商品が1点ある        ← canonical（現在の仕様）
   + カート内に商品が1点以上ある     ← 最新の観測（コード側）
   ```

3. 差分の内容から対応を判断する:

   **a. コード側が正しい場合（仕様変更の追従）:**
   1. テストケースの Given/When/Then を新しい挙動に合わせて編集する
   2. accept-fingerprint 操作で新しい指紋を受け入れる
   3. drift が解消される

   > API: `POST /api/v1/projects/:pid/testcases/:id/accept-fingerprint`（OCC 必須）

   **b. コード側がバグの場合:**
   1. drift はそのまま残す
   2. `[※未実装]` テストケースにコメント・メモを残す（「#123 のバグにより乖離」）
   3. 開発者にバグ報告し、修正後に再同期で drift が自動解消されることを待つ

   **c. 意図的な仕様変更で、テスト仕様も更新が必要な場合:**
   1. テストケースを編集して新仕様に合わせる
   2. ステータスを一旦 `draft` に戻す（approved → draft の差し戻し）
   3. 再レビュー後に `approved` に戻す

### UC-16 stale（未観測）テストケースの検知と対応

Discovery の同期で出現しなくなったテストケース。コード側でテストが削除・リネームされた可能性がある。

**UIでの見え方:**
- テストケース一覧の該当行に `stale` バッジが表示される
- テストケース詳細でどのオリジン（衛星）が stale かをドリルダウン表示

**対応フロー:**

1. テストケース一覧で `stale` バッジのあるテストケースを確認する
2. `[※未実装]` 最終観測日時を確認し、いつから見えなくなったかを把握する
3. 対応を判断する:

   **a. テストが不要になった場合（コード削除済み）:**
   1. テストケースを `archived` に変更する
   2. テストケースは物理削除されず、履歴として残る

   **b. リネーム・移動された場合:**
   1. 新しいテストケースが同時に取り込まれていないか確認する
   2. `[※未実装]` 旧テストケースと新テストケースの内容を並べて比較する
   3. 旧テストケースを `archived` にする
   4. 新テストケースに必要な編集を加えて承認する

   **c. 一時的な除外の場合（ブランチ切替等）:**
   1. そのまま放置する
   2. 次の同期で再出現すれば stale フラグは自動解除される

> `approved` のテストケースは stale 化しない。人間が承認した仕様は、コード側で消えても自動的には無効化されない。

### UC-17 新規テストケースの追加確認

再同期で新しいテストケースが取り込まれた場合の確認。

1. `[※未実装]` 同期結果の通知（新規 N 件、drift M 件、stale K 件）を確認する
2. テストケース一覧でステータス `draft` + 所有権 `machine` でフィルタ（`status=draft&ownership=machine`）
3. `[※未実装]` 作成日でソートし、直近の取り込み分を特定する
4. フェーズ 3 のレビュー・承認フローに進む

## フェーズ 5：テストの種別と管理

### テストケースのカテゴリ

TMS は各テストケースを以下の 4 カテゴリで分類する。Discovery の自動判定か、手動登録時に人間が指定する。

| カテゴリ | 検証内容 | 例 |
|---|---|---|
| **正常系** (normal) | 正常な入力・操作で期待結果が得られること | 有効なカードで決済が成功する |
| **異常系** (abnormal) | 不正な入力・異常な操作時の振る舞い | 無効なカードで決済エラーが返る |
| **境界値** (boundary) | 入力値の境界における振る舞い | 金額 0 円、上限額、上限 + 1 円 |
| **エラーハンドリング** (error_handling) | システムエラー・外部障害時の振る舞い | DB 接続失敗時にリトライ後にフォールバック |

### テストレベルの管理 `[※未実装]`

上記のカテゴリはテストの「性質」を表すが、テストの「レベル」（単体・結合・E2E 等）は別の軸である。現スペックにはテストレベルの分類方法が未定義。

`[※未実装]` 想定される管理方法:

| 方法 | 概要 | 利点 | 欠点 |
|---|---|---|---|
| metadata タグ | `{"tags": ["unit"]}` 等をタグで管理 | 柔軟。すぐ使える | 表記揺れの可能性 |
| プロジェクト分割 | レベルごとにプロジェクトを分ける | 明確な分離 | プロジェクト数が増大 |
| 専用フィールド追加 | `test_level` 列を追加 | 型安全。フィルタ容易 | スキーマ変更が必要 |

> MVP では metadata タグで柔軟に対応し、実運用のフィードバックで構造化を検討する。

### 各テスト種別の登録・管理パターン

#### 単体テスト（Unit Test）

- **主な登録方法:** Discovery による自動取り込み
- **target:** メソッド単位（例：`com.example.PaymentService#charge`）
- **管理方針:** 数が多い。machine 所有のまま観測を蓄積し、重要なテストだけ人間がレビュー・承認して human 化する
- **drift 対応:** 頻繁に発生しうるが、machine 所有のものは自動更新されるため、承認済みのもののみ対応

#### 結合テスト（Integration Test）

- **主な登録方法:** 自動取り込み + 手動補完
- **target:** サービス間の連携ポイント、API エンドポイント
- **管理方針:** サービス境界を跨ぐため、人間による承認を重視。全件レビューが望ましい
- **特記:** 複数サービスに跨る場合、どのプロジェクトに属するかの判断が必要

#### E2E テスト（End-to-End Test）

- **主な登録方法:** 手動登録が中心（コード解析では抽出しにくい）
- **target:** ユーザーシナリオ、画面フロー
- **管理方針:** ビジネス要件との対応が重要。少数だが1件1件が重い。全て human 所有で丁寧に管理
- **特記:** Given/When/Then がユーザーストーリーに近い形になる

#### 回帰テスト（Regression Test）

- **主な登録方法:** バグ修正時に手動登録
- **target:** バグが発生した機能・メソッド
- **管理方針:** バグチケットへの参照を metadata に記録（例：`{"tags": ["regression"], "bug_ticket": "PROJ-123"}`）
- **特記:** 「このバグが再発しないこと」を検証する仕様なので、archived にしにくい

#### 性能テスト（Performance Test）`[※未実装]`

- **現状の課題:** カテゴリ（normal/abnormal/boundary/error_handling）のいずれにも当てはまらない
- **暫定対応:** カテゴリは `normal` にし、metadata タグで `performance` を付与
- **将来:** カテゴリの拡張、または別軸での分類を検討

## フェーズ 6：日常運用

### UC-18 テストケースの検索・フィルタ

テストケース一覧画面で目的のテストケースを探す。

**設計スペック定義済みのフィルタ:**
- ステータス（draft / approved / archived）
- カテゴリ（normal / abnormal / boundary / error_handling）
- 所有権（machine / human）
- drift あり / なし
- stale あり / なし
- target（対象クラス・メソッド名での部分一致）

**`[※未実装]` 追加が望ましいフィルタ:**
- メタデータのタグ
- フリーテキスト検索（タイトル、Given/When/Then の内容）
- 作成日・更新日の範囲指定

### UC-19 開発者が自分のコードのテスト状況を確認する

開発者が「自分が変更したクラスに対するテストがどれだけ存在し、どういう状態か」を確認する。

1. テストケース一覧を開く
2. target フィルタに自分のクラス名を入力する（例：`PaymentService`）
3. 該当するテストケースの一覧が表示される
4. ステータス別の内訳を確認する:
   - `approved`: 承認済み。テストコード生成の対象
   - `draft`: 未承認。レビューが必要
   - drift あり: コード変更による乖離。確認が必要

> API: テストケース一覧の `target` クエリパラメータ（部分一致検索）で実現。

### UC-20 Gherkin ビューの利用

テストケースを Gherkin 形式で表示する。テスト仕様書としてステークホルダーに共有する場面で使う。

1. テストケース詳細画面で「Gherkin ビュー」タブを選択
2. Given/When/Then/Parameters が Gherkin 形式で整形表示される

```gherkin
Feature: 決済処理

  Scenario: 有効期限切れカードで決済を試みるとエラーが返る
    Given 有効期限が過去のカード情報が登録されている
    When そのカードで 1,000 円の決済を実行する
    Then 決済が拒否され、エラーコード CARD_EXPIRED が返る
```

3. `[※未実装]` Gherkin テキストをクリップボードにコピー、またはファイルとしてエクスポート

### UC-21 変更履歴の確認

テストケースの変更経緯を追跡する。「誰がいつ何を変えたか」を知る。

1. テストケース詳細画面で「履歴」タブを開く
2. 変更の時系列が新しい順に表示される:
   - **日時:** いつ変更されたか
   - **実行者:** 誰が変更したか（ユーザー名 or 衛星トークン名）
   - **操作種別:** created / updated / status_changed / imported
   - **差分:** 変更されたフィールドの before/after

**例:**
```
2026-06-28 14:30  田中 (editor)  status_changed
  status: draft → approved

2026-06-28 14:25  田中 (editor)  updated
  then: "エラーが返る" → "エラーコード CARD_EXPIRED が返る"
  category: normal → abnormal

2026-06-28 10:00  token:discovery-ci  imported
  (初回取り込み)
```

### UC-22 ダッシュボード `[※未実装]`

プロジェクト全体のテスト管理状況を一目で把握する。QA リード・PM の主な利用画面。

`[※未実装]` 表示項目（案）:
- テストケース総数（ステータス別の内訳グラフ）
- 承認率（approved / 全体）
- 未レビュー件数（draft + machine 所有）
- drift 発生中の件数
- stale の件数
- 直近の変更アクティビティ（最近編集・承認されたテストケース）
- カテゴリ別の分布

### UC-23 レポート・エクスポート `[※未実装]`

テスト管理の状況を外部に共有する。

`[※未実装]` 機能（案）:
- テストケース一覧の CSV エクスポート
- ステータス別・カテゴリ別の集計表
- `[※未実装]` 期間指定の変更推移（新規登録数、承認数、stale 発生数の推移）

## フェーズ 7：保守・運用管理

### UC-24 API トークンのローテーション

セキュリティのため、定期的にトークンを更新する。

**実行者:** 管理者

1. 新しいトークンを発行する（UC-06）
2. 衛星サービスの設定を新トークンに更新する
3. 新トークンで同期が正常に動作することを確認する
4. 古いトークンを失効させる（トークン一覧画面から「失効」を実行）
5. 失効済みトークンは認証時に即座に拒否される

### UC-25 ユーザーのロール変更

**実行者:** 管理者

1. ユーザー管理画面で対象ユーザーを選択
2. ロールを変更する（例：viewer → editor）
3. 対象ユーザーの既存セッションが全て無効化される
4. 次回ログイン時から新ロールで動作する

> API: `PATCH /api/v1/users/:id` — ロール変更時は全セッション無効化。

### UC-26 パスワード変更

**実行者:** 本人

1. プロフィール画面からパスワード変更を選択
2. 現在のパスワードと新しいパスワードを入力
3. パスワードが更新され、他の全セッションが無効化される

> API: `PATCH /api/v1/auth/password` — 現在のパスワード検証 + 他セッション無効化。

### UC-27 パスワードリセット `[※未実装]`

パスワードを忘れた場合の回復手段。

**方法 A: 管理者による手動リセット（MVP 向け）**
1. ユーザーが管理者に連絡する
2. 管理者がユーザー管理画面で一時パスワードを設定する
3. ユーザーが一時パスワードでログインし、パスワードを変更する

> API: `POST /api/v1/users/:id/reset-password`（admin 限定）

**方法 B: セルフサービスリセット（将来）**
1. `[※未実装]` ログイン画面で「パスワードを忘れた場合」を選択
2. `[※未実装]` メールアドレスを入力
3. `[※未実装]` リセット用の一時リンクがメールで送信される
4. `[※未実装]` リンクから新しいパスワードを設定する

> 方法 B にはメール送信基盤が必要。MVP の小規模運用では方法 A で十分。

### UC-28 テストケースのアーカイブ

不要になったテストケースをアーカイブする。物理削除は行わない。

**実行者:** 編集者

1. テストケース詳細画面でステータスを `archived` に変更する
2. テストケースは一覧画面のデフォルト表示から消える（archived フィルタで表示可）
3. 以後、Discovery が同期で同じテストを検出しても、archived のステータスは変わらない
4. 変更履歴は保持される

**アーカイブからの復帰:**
1. archived のテストケースを開く
2. 「復帰」操作でステータスを `draft` に戻す（archived → draft）
3. 必要に応じて再レビュー・再承認する

> API: `PATCH /api/v1/projects/:pid/testcases/:id` で `status: "draft"` に変更、または一括操作の `restore` アクション。

### UC-29 プロジェクトの追加

新しいサービスやリポジトリのテスト管理を開始する。

1. UC-04 でプロジェクトを作成する
2. UC-06 で API トークンを発行する
3. Discovery を新プロジェクトに設定して実行する
4. UC-07 〜 UC-11 のフローでレビュー・承認を進める

## フェーズ 8：テストコード生成との連携（将来）

### UC-30 承認済みテストケースからのテストコード生成

**前提:** Test Generation 衛星サービスが稼働していること（MVP スコープ外）

1. QA エンジニアがテストケースを `approved` にする
2. Test Generation 衛星が TMS API で approved テストケースを取得する
3. テストケースの Given/When/Then/Parameters からテストコード（JUnit 等）を自動生成する
4. 生成されたコードが PR としてリポジトリに提出される
5. `[※未実装]` TMS 上で生成結果（成功/失敗、カバレッジ等）が確認できる

### UC-31 CI 失敗時の自動修復

**前提:** Self-Healing 衛星サービスが稼働していること（MVP スコープ外）

1. CI でテストが失敗する
2. Self-Healing 衛星が失敗原因を分析する
3. テストコードの修正 PR を自動生成する
4. 修正がマージされると、Discovery が再同期する
5. TMS 上のテストケースが最新状態に反映される

## テストケースのライフサイクル（まとめ）

```
                        自動取り込み (Discovery)
                              │
                              ▼
[手動作成] ──────▶ draft / human    draft / machine
                       │                │
                       │         編集 or 承認で
                       │         machine → human
                       │                │
                       ├────────────────┘
                       │
                  レビュー・編集
                       │
                       ▼
                 approved / human ──── 再レビュー必要 ──▶ draft / human
                       │                                      ▲
                  テスト不要                                   │
                       │                                  復帰操作
                       ▼                                      │
                 archived / human ────────────────────────────┘
```

**直交フラグ（ステータスとは独立）:**
- `drift`: human 所有テストケースのコード側挙動との乖離。approved でも draft でも発生しうる
- `stale`: Discovery の同期で未観測。approved には適用されない
